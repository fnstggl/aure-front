Technical memo · July 2026

Choose the best fleet trajectory, not just the best next action.

One live fleet state. Many counterfactual futures. One safe trajectory.

Kubernetes, Slurm, serving schedulers, and autoscalers already make and execute control decisions well. Aurelius adds a pre-execution simulation layer above them. Before decisions execute, it forks the live fleet state into competing simulated futures, each applying a different coupled control strategy across batching, routing, placement, capacity, prewarming, admission, migration, precision, and clock.

Each trajectory advances queues, topology pressure, warm state, capacity, SLA risk, and cost through time. Trajectories that violate hard SLA, capacity, power, or policy constraints are rejected. The highest-value feasible trajectory is returned to the existing control plane, which stays authoritative and executes.

AURELIUS Technical memo · 03

One fleet. Many futures. Simulated before execution.

Each trajectory advances queues, topology pressure, warm state, capacity, SLA risk, and cost through t+1, t+2, t+3. Every future runs read-only: candidate evaluation never touches the live fleet.

1
Mirror the fleet.

A persistent world model carries capacity, placement, warm state, queues, migrations in flight, and the operator cost basis, advanced period by period.

2
Generate competing coupled trajectories.

Physics-guided generation proposes a bounded set of coupled strategies across the control surfaces, never one knob at a time. Known-strong candidates are always retained.

3
Run each future forward, read-only.

Every candidate trajectory is simulated against forecast conditions using only information available at the decision point. No future truth leaks in.

4
Reject unsafe futures. Return the winner.

Hard SLA, capacity, power, and policy gates eliminate trajectories outright. The best feasible future is recommended, with a deterministic fallback when confidence is low.

Aurelius sits above Kubernetes, Slurm, serving schedulers, routing systems, autoscalers, and fleet-management systems. It coordinates their decisions against a common anticipated future. It does not replace them. It reads scheduler metadata only, never payloads, model outputs, or weights.

AURELIUS Technical memo · 04

A signal too large to ignore and too simulated to trust without operator data.

SLA-safe goodput per dollar vs the constructed production-class composite. Same replayed load per window, 1,549,249 requests total, 84.5% fewer GPU-hours, better SLA in all three windows.

What this establishes

Predictive coupled planning found substantially more efficient safe trajectories than the constructed public-replay composite under the selected uncapped windows.

What this does not establish

Expected performance against a particular fleet or internal scheduler. The result is simulated and load-dependent, and its magnitude is sensitive partly to modeled precision and batching behavior.

Test one bounded historical window.

Input

Anonymized scheduler metadata from one representative workload or cluster window. No customer identity, payloads, model outputs, weights, production actuation, or live integration.

Process

Reconstruct the historical baseline and evaluate Aurelius counterfactual trajectories under identical recorded conditions, with a protocol fixed before any data moves.

Output

Matched SLA, GPU-hour, utilization, and economic comparisons; simulator-fidelity checks; sensitivity analysis; failure cases and negative findings; and a verdict on whether shadow mode is warranted.

The ask

Identify one technically appropriate owner and one bounded historical dataset on which this can be tested.
