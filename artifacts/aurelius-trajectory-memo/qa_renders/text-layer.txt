Kubernetes, Slurm, serving schedulers, and autoscalers
already make and execute control decisions well. Aurelius
adds a pre-execution simulation layer above them. Before
decisions execute, it forks the live fleet state into
competing simulated futures, each applying a different
coupled control strategy across batching, routing,
placement, capacity, prewarming, admission, migration,
precision, and clock.
Each trajectory advances queues, topology pressure,
warm state, capacity, SLA risk, and cost through time.
Trajectories that violate hard SLA, capacity, power, or
policy constraints are rejected. The highest-value feasible
trajectory is returned to the existing control plane, which
stays authoritative and executes.

The winning strategy combined lower precision, aggressive batching, and a higher clock. No isolated-knob
search represented that future: tuned one surface at a time, each move looked worse, so the bundle was never
proposed. Once the coupled trajectory entered the candidate set, the unchanged simulator selected it.
The value was not hidden in a better score. It was hidden in a
future the previous search never considered.
These three numbers are the controlled experiment: Benchmark V1, the frozen cap-controlled harness, primary market window,
the only regime where exhaustive enumeration is tractable and search regret can be measured exactly, at roughly 40 trajectory
evaluations per decision.
The 8.24× headline on pages 1 and 4 is the same coupled search run separately at full uncapped load against the production-class
composite. The two benchmarks use different baselines and loads and are never mixed. Complete methodology in the full
technical report.

Aurelius sits above Kubernetes, Slurm, serving schedulers, routing systems, autoscalers, and fleet-management systems. It
coordinates their decisions against a common anticipated future. It does not replace them. It reads scheduler metadata only,
never payloads, model outputs, or weights.

What this establishes
Predictive coupled planning found substantially more
efficient safe trajectories than the constructed public-replay
composite under the selected uncapped windows.
What this does not establish
Expected performance against a particular fleet or internal
scheduler. The result is simulated and load-dependent, and
its magnitude is sensitive partly to modeled precision and
batching behavior.
Test one bounded historical window.
Input
Anonymized scheduler metadata from one representative workload or cluster window. No customer
identity, payloads, model outputs, weights, production actuation, or live integration.
Process
Reconstruct the historical baseline and evaluate Aurelius counterfactual trajectories under identical
recorded conditions, with a protocol fixed before any data moves.
Output
Matched SLA, GPU-hour, utilization, and economic comparisons; simulator-fidelity checks; sensitivity
analysis; failure cases and negative findings; and a verdict on whether shadow mode is warranted.
The ask
Identify one technically appropriate owner and one bounded historical dataset on
which this can be tested.
